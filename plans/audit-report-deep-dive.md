## Category 1 — Correctness / Bugs

### [CORRECT-01] ViT conv-stem downsampling diverges from `patch_size` for non-power-of-2 patches

- **Evidence**:
  - `src/models/backbones/vit.py:83-91` — the final stem layer's stride is computed as
    ```python
    remaining_stride = max(1, patch_size // (2 ** len(stem_channels)))
    layers += [
        nn.Conv2d(c_in, embed_dim, kernel_size=remaining_stride + 1,
                  stride=remaining_stride, padding=remaining_stride // 2, bias=False),
        nn.BatchNorm2d(embed_dim),
    ]
    ```
    The `//` is **integer floor division**. Total stem downsampling is
    `(2 ** len(stem_channels)) * remaining_stride`, which only equals `patch_size`
    when `patch_size` is divisible by `2 ** len(stem_channels)`.
  - `src/models/backbones/vit.py:10-15` — the module docstring advertises four base
    models including `swin_tiny_patch4_window7_224` (patch=4) and ViT variants
    where patch can be 14 or 32.
- **Reproduction** (computed during audit):
  - `patch_size=16`, `stem_channels=(32,64,128)` (len 3): `2^3=8`, `remaining=16//8=2`, total `8*2=16` → ✓ matches
  - `patch_size=14`: `remaining=14//8=1`, total `8*1=8` → ✗ token grid is ~2× too large
  - `patch_size=32`: `remaining=32//8=4`, total `8*4=32` → ✓ matches
- **Impact**:
  For the default `vit_small_patch16_384` config this is **correct** — the bug is
  latent. But the moment someone selects a `/14` patch model (common in modern
  ViT/DeiT derivatives) or the advertised Swin config, the stem produces a token
  grid that doesn't match the pretrained positional embedding's grid. The
  `_interpolate_pos_embed` fallback (`vit.py:208-233`) will silently bicubic-interpolate
  the pos-embed to the wrong grid size, and the model will run without error but
  with degraded spatial priors — a silent accuracy regression that's very hard
  to diagnose.
- **Root cause**: assuming `patch_size` is always divisible by `2 ** len(stem)`,
  encoded as floor division.
- **Effort**: M. The fix must (a) make the final-layer stride produce an *exact*
  total downsample of `patch_size` (not floor-approximate it), and (b) add an
  assertion + test that `stem(x)` of a `(1,3,H,H)` tensor yields a feature map
  of spatial size `H/patch_size` for every supported backbone.
- **Risk**: MED. Changing stem geometry changes the feature space; any partially-
  trained checkpoint would be invalidated. Since no checkpoints exist yet, the
  blast radius is contained to "fix it before the first real run."
- **Confidence**: HIGH — the arithmetic is reproduced above.
- **Fix sketch**: Require `2 ** len(stem_channels)` to divide `patch_size`
  (raise `ValueError` otherwise), OR compute the final stride as
  `patch_size // (2 ** len(stem_channels))` but validate
  `(2 ** len(stem_channels)) * remaining_stride == patch_size` and raise if not
  (forcing the user to pick a compatible stem depth). Add a unit test that, for
  each advertised backbone name, asserts the stem's output spatial size equals
  `input_size / patch_size`.

### [CORRECT-02] Threshold selection assumes a specific ROC-threshold ordering that can misreport the WHO operating point

- **Evidence**:
  - `src/evaluation/metrics.py:178-191` — `find_who_threshold`:
    ```python
    fpr, tpr, thresholds = roc_curve(yt, yp)
    valid_idx = np.where(tpr >= sensitivity_target)[0]
    if len(valid_idx) == 0:
        idx = int(np.argmax(tpr))
    else:
        idx = int(valid_idx[0])  # highest threshold that hits the target
    chosen_threshold = float(thresholds[idx])
    ```
  - `src/evaluation/calibration.py:126-130` — same pattern inside
    `PlattCalibrator._find_threshold`.
  - The comment claims `valid_idx[0]` is the "highest threshold still meeting
    the target." `sklearn.metrics.roc_curve` returns `thresholds` in
    **descending** order with `tpr` **non-decreasing** — so `valid_idx[0]` is
    indeed the *first* (highest-threshold) index where `tpr >= target`. That
    part is correct.
- **The actual hazard** (why this is MED not a clear bug):
  `roc_curve`'s first threshold is `max(y_score) + 1` (a sentinel) with
  `tpr=0, fpr=0`. If the sentinel happens to be the only "valid" entry in an
  edge case, or if `sensitivity_target` is met only trivially, the returned
  threshold can be `+inf`-adjacent and unusable. More importantly, the *specificity*
  reported at that index uses `1 - fpr[idx]`, but `roc_curve` collapses ties —
  multiple distinct raw thresholds can map to the same `(fpr, tpr)` point, and
  the one `roc_curve` returns is an arbitrary representative. For a **clinical
  WHO-TTP report**, the chosen operating threshold is the single number that
  drives the "positive/negative" decision; an arbitrary tiebreak is a real
  correctness concern even if rare.
- **Impact**: For well-separated scores (the common case) this is fine. For
  near-TPP-boundary performance — exactly the regime this system is designed
  for — the reported threshold can be a non-robust tiebreak, and small data
  perturbations will move it. This undermines the per-site calibration story
  (`calibration.py`), which is the project's central deployment claim.
- **Effort**: S. Replace the implicit ordering assumption with an explicit
  threshold search over the **unique sorted raw scores** (not the collapsed
  ROC thresholds), picking the highest score value that still achieves
  `sensitivity >= target`.
- **Risk**: LOW. Pure computation change; well-covered by a test with a
  constructed tie case.
- **Confidence**: MED — the common path is correct; the edge case is real but
  its frequency depends on score distributions not yet observed (no trained
  model exists).
- **Fix sketch**: Compute `sensitivity` and `specificity` directly at each
  unique candidate threshold (`np.unique(y_score)`), select the highest
  threshold with `sens >= target`, and assert `spec >= 0.70` explicitly. Add a
  test with deliberately tied scores around the target sensitivity.

pipeline with no network surface, no auth layer, no user input crossing into
interpreters, and no credentials in the committed tree (verified: no `.env`,

patterns returned nothing).

The only security-adjacent notes:

- **No secret hygiene risk currently**, but `config.py:25-26` writes
  `data/ checkpoints/ logs/ outputs/` into existence at *import time*. Once
  training runs, these dirs will contain model weights and potentially PHI
  (image-derived features). With no `.gitignore` (see DX-01), the next
  `git add .` would stage them. This is folded into the `.gitignore` finding
  rather than reported separately — the fix is the same.
- **`torch.load` is used without `weights_only=True`** in four places:
  `cnn.py:162`, `vit.py:268`, `segmentation.py:313`, and `scripts/evaluate.py:77`
  / `scripts/deploy.py:47`. `torch.load` on an untrusted `.pt` file is a known
  arbitrary-code-execution vector (pickle deserialization). For a research
  repo loading only self-produced checkpoints this is LOW risk, but since
  torch ≥ 2.6 the safe default is `weights_only=True`; passing it explicitly is
  cheap defense-in-depth and worth a one-line note in any plan that touches
  checkpoint loading.

### [DEBT-03] Two divergent `get_parameter_groups` implementations; one is dead code

- **Evidence**:
  - `src/models/backbones/cnn.py:211-229` — `get_parameter_groups(model, backbone_lr, head_lr)`

  - `src/models/__init__.py:226-244` — `_TBEnsemble.get_parameter_groups` re-implements
    the same logic but keys off `self.cnn.parameters() + self.vit.parameters()`.
  - `src/training/scheduler.py:89-93` — `build_optimizer` calls

- **Impact**: Dead code that will mislead. A future reader sees two
  implementations of the same idea and doesn't know which is authoritative.
  The CNN-only version is also subtly different (it would put the ViT into the

- **Effort**: S. Delete `cnn.py:211-229` and its docstring.
- **Risk**: LOW — it's dead code.
- **Confidence**: HIGH.
- **Fix sketch**: Remove the standalone `get_parameter_groups` from `cnn.py`;
  the ensemble method is the single source of truth.

### [DEBT-04] Ensemble `get_parameter_groups` keys off `id(param)` — fragile under parameter sharing

- **Evidence**: `src/models/__init__.py:230-244`:
  ```python
  backbone_ids = {id(p) for p in list(self.cnn.parameters()) + list(self.vit.parameters())}
  backbone_params = [p for p in self.parameters() if id(p) in backbone_ids]
  head_params     = [p for p in self.parameters() if id(p) not in backbone_ids]
  ```
- **Impact**: This is *correct today* — each parameter object appears exactly
  once in `.parameters()`, so `id()`-set membership cleanly partitions
  backbone vs. head. The fragility is forward-looking: if the fusion module
  ever *reuses* a backbone parameter (e.g. a tied weight), that param would be
  in the backbone set and get the backbone LR even when used by the head. Not
  a bug now; a foot-gun for a future architecture change.
- **Effort**: S. Prefer walking named submodules (`self.cnn`, `self.vit` →
  backbone; everything else → head) and collecting params by reference from
  the submodule tree, which is robust to sharing.
- **Risk**: LOW.
- **Confidence**: MED — correct today, the hazard is prospective.

### [DEBT-05] `scale_to_cad4tb` is linear; docstring claims CAD4TB convention

- **Evidence**: `src/models/heads.py:132-144`:
  ```python
  def scale_to_cad4tb(prob): ... return prob * 100.0
  ```
  Docstring: *"Convert a scalar TB probability in [0, 1] to a 0–100 integer
  score matching the CAD4TB clinical display convention."* The README
  architecture diagram and `TBPrediction` docstring also reference "CAD4TB-style
  0–100 score."
- **Impact**: CAD4TB's actual 0–100 score is a **monotonic but non-linear**
  mapping derived from its own calibrated logits; it is not `prob*100`. For a
  research clone this is probably an acceptable simplification, but the
  *claim* in the docstring is unverified and, for a SaMD-adjacent project,
  the scoring convention matters: clinicians comparing this tool's output to
  CAD4TB side-by-side would see different numbers for the same patient.
- **Effort**: S to *fix the docstring* (downgrade the claim to "linear 0–100
  rescaling"); L to *implement the real CAD4TB mapping* (requires the CAD4TB
  scoring spec, which may not be public).
- **Risk**: LOW for the docstring fix.
- **Confidence**: MED — the linear-vs-nonlinear distinction is real; whether
  it matters for this project is a product call.
- **Fix sketch**: Either (a) rename to `scale_to_100` and soften the docstring
  to "linear rescale to 0–100," or (b) if CAD4TB fidelity is a requirement,
  document the gap as a known limitation and open a direction item to source
  the real mapping.

## Category 9 — Direction (features & where to take this next)

These are **options for the maintainer**, not bugs. Each is grounded in
specific repo evidence.

### [DIR-01] Wire Grad-CAM into the eval and deploy artifacts (highest product value)

- **Grounding**: The README states "Grad-CAM on every positive — Mandatory for
  clinical trust (WHO-evaluated CAD standard)" (`README.md:187`) and the
  architecture diagram shows "Score 0–100 + Grad-CAM overlay"
  (`README.md:33`). The code has a complete, working `GradCAM` class
  (`src/models/interpretability.py:52-232`) and `TBEnsemble.enable_gradcam()`
  (`src/models/__init__.py:128-141`). But:
  - `scripts/evaluate.py:160-174` — the `--audit-fn` flag's Grad-CAM loop is a
    **comment**, not code: *"Full Grad-CAM audit would reload each image and
    run the CAM — left as a post-processing step for the reporting pipeline."*
  - `src/deployment/inference.py:354` — `InferenceEngine.predict` always returns
    `gradcam=None`, ignoring the documented `result.gradcam` field.
- **Value**: Directly fulfills the single most prominent clinical-trust claim
  in the README. For a SaMD triage tool, "every positive has a heatmap" is the
  feature that distinguishes a deployable system from a notebook.
- **Trade-off**: The eval-side wiring is straightforward (the model already
  supports `return_gradcam=True`). The deploy side is harder because ONNX
  export currently strips Grad-CAM (`export.py:92-94` calls
  `model.disable_gradcam()`), so deploy-time CAM needs either a separate
  torch-backed CAM path or an ONNX-compatible gradient workaround.
- **Coarse effort**: M (eval side) + L (deploy side). Scope as a design/spike
  plan first: define the CAM-output contract, decide torch-vs-ONNX, then build.

### [DIR-02] Commit to or delete the hand-rolled MoCo-v3 loop

- **Grounding**: `src/training/pretrain.py` implements a full MoCo-v3 training
  loop (~240 lines), but its own module docstring says: *"Full MoCo-v3 / DINO
  implementations are in the `lightly` or `solo-learn` libraries. This module
  wraps them..."* (`pretrain.py:16-19`) and *"For large-scale runs (>100k
  images) we recommend the `solo-learn` library"* (`pretrain.py:59-63`). The
  loop is untested (see TEST-01) and its checkpoint-format contract with
  `build_cnn_backbone(pretrained="mocov3_cxr")` is asserted but unverified.
- **Value**: Resolves the half-state. Either (a) the in-repo loop is the
  supported path → test it and document it as primary, or (b) it's a reference
  impl → mark it clearly as such and make `solo-learn`/`lightly` the
  documented path with a conversion script.
- **Trade-off**: Option (a) commits to maintaining a contrastive-learning
  loop (a moving target). Option (b) reduces surface area but adds a
  dependency-on-external-tool story to the docs.
- **Coarse effort**: M either way. Scope as a direction plan that picks the
  option and specifies the test/doc/conversion work.

### [DIR-03] Establish the dataset-prep → CSV contract as a first-class script

- **Grounding**: The README documents a precise CSV schema
  (`image_path,tb_label,findings_label,active_inactive_label,site,view_position,split,mask_path`,
  `README.md:170-173`) and `DATASETS.md` (22 KB) exists describing datasets,
  but there is **no script** that converts any raw dataset (Shenzhen,
  Montgomery, TBX11K, etc.) into that CSV format. `TBDataset._load_csv`
  (`src/data/dataset.py:152-166`) is the only consumer. The recent git log
  (`feat: use official TBX11K from dataset-ninja`, `fix: correct dataset paths`)
  shows dataset work is actively happening — but in the notebook, not as a
  reusable script.
- **Value**: The pipeline cannot run end-to-end without these CSVs; right now
  producing them is undocumented manual work. A `scripts/prepare_dataset.py`
  per source dataset is the missing on-ramp to actually training.
- **Trade-off**: Each dataset has its own layout/label convention; this is
  N scripts, not one. Start with the two NIH benchmarks (Shenzhen/Montgomery)
  since they're smallest and most standard.
- **Coarse effort**: M per dataset.
